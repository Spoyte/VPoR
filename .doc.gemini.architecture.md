# VPoR Architecture & EVVM Configuration

## 1. System Overview
The VPoR system is composed of three main layers:
1.  **The Ledger (VPoR Chain):** A custom, permissioned EVVM instance running on Sepolia.
2.  **The Verifier (Chainlink):** A decentralized oracle network that fetches real-world asset data.
3.  **The Application (Frontend/Backend):** The user interface and the backend services that drive the updates.

```mermaid
graph TD
    User[User] -->|Connects Wallet| Frontend[VPoR Dashboard]
    Frontend -->|Reads State| EVVM[VPoR Chain (EVVM)]
    Frontend -->|Decrypts Proof| Local[Local Browser Storage]
    
    Backend[Exchange Backend] -->|Submits Liability Merkle Root| EVVM
    Backend -->|Generates Proofs| Merkle[Merkle Sum Tree]
    
    Chainlink[Chainlink Network] -->|Fetches Asset Data| BTC[Bitcoin Network / Bank API]
    Chainlink -->|Updates Assets| Bridge[ChainlinkEVVMBridge (Sepolia)]
    Bridge -->|Executes Transaction| EVVM
```

## 2. The VPoR Chain (EVVM)
We are deploying a **Permissioned Virtual Blockchain** using the MATE stack.

### Why EVVM?
-   **Sovereignty:** We control the gas logic and access rules.
-   **Throughput:** **Async Nonces** allow us to process thousands of liability updates in parallel, which is impossible on a standard sequential-nonce chain.
-   **Cost:** We can subsidize the gas for our own updates, making verification free for users.

### Configuration
-   **Host Chain:** Sepolia Testnet.
-   **Execution Mode:** Permissioned (Whitelist).
-   **Gas Token:** Native VPoR Token (or Gasless for Whitelisted Executors).

### Access Control (The Whitelist)
The core security model relies on restricting **Write Access** to only two entities:
1.  **The Exchange (Liability Publisher):** Authorized to call `submitLiabilityProof`.
2.  **The Chainlink Bridge (Asset Publisher):** Authorized to call `updateVerifiedAssets`.

*All other addresses have Read-Only access.*

## 3. Data Structures

### Liability Merkle Sum Tree
To prove solvency without revealing individual balances, we use a Merkle Sum Tree.
-   **Leaf Node:** `Hash(UserAddress, Balance, Nonce)` + `Balance`
-   **Branch Node:** `Hash(LeftChild, RightChild)` + `Sum(LeftBalance, RightBalance)`
-   **Root:** `RootHash` + `TotalLiabilities`

### Privacy (ECIES)
We do not store plain text balances on-chain.
-   **On-Chain:** We emit an event `ProofOfInclusion(address user, bytes encryptedPayload)`.
-   **Off-Chain:** The user's wallet (Metamask) holds the private key.
-   **Decryption:** The frontend requests `eth_decrypt(encryptedPayload)` to reveal the balance and Merkle Proof to the user.

## 4. Async Nonces
Standard Ethereum transactions must be ordered sequentially (Nonce 1, 2, 3...). If Nonce 2 fails, Nonce 3 is stuck.
**MATE's Async Nonces** allow us to send Nonce 1, 2, and 3 simultaneously. They are processed in parallel.
-   **Use Case:** The Exchange Backend splits the user base into 10 shards.
-   **Execution:** It submits 10 parallel transactions to update the liabilities for each shard.
-   **Result:** 10x throughput compared to a standard rollup.
