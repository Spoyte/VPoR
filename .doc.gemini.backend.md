# VPoR Backend & Scripting Specifications

## 1. Overview
The Backend acts as the "Exchange" in our simulation. It is responsible for the heavy lifting: generating proofs, encrypting data, and submitting transactions to the VPoR Chain.

## 2. The Liability Generator
**Goal:** Create a deterministic dataset of user liabilities.

### Data Structure (`users.json`)
```json
[
  {
    "id": "user_1",
    "address": "0xUserWallet...",
    "balance": "5500000000000000000", // 5.5 ETH
    "nonce": 1
  },
  ...
]
```

### Merkle Sum Tree Logic
We need a custom Merkle Tree that sums values up the tree.
-   **Library:** `merkletreejs` (extended) or a custom implementation.
-   **Leaf:** `keccak256(address + balance + nonce)`
-   **Proof:** Standard Merkle Proof + Sibling Sums.

## 3. The ECIES Encryptor
**Goal:** Privacy. Only the user should see their data.

### Encryption Flow
1.  **Input:** User's Public Key (derived from their address or known beforehand for the demo).
    *   *Hackathon Shortcut:* For the demo, we might use a fixed set of test wallets where we know the keys, or ask users to "Register" their public key first.
2.  **Payload:**
    ```json
    {
      "balance": "5.5 ETH",
      "proof": [...],
      "root": "0x...",
      "timestamp": 1234567890
    }
    ```
3.  **Algorithm:** ECIES (Elliptic Curve Integrated Encryption Scheme).
4.  **Output:** Encrypted Bytes.

## 4. The Executor Script ("The Fisher")
**Goal:** Submit the data to the chain.

### Logic
1.  **Load Data:** Read `users.json`.
2.  **Build Tree:** Generate the Merkle Sum Tree.
3.  **Encrypt:** Generate the encrypted payload for each user.
4.  **Batch:** Group users into batches (e.g., 100 per tx) if necessary, or just submit the Root and emit events.
    *   *Optimization:* We only need to submit the **Root** to the contract storage. The **Proofs** are emitted as events.
5.  **Submit:** Call `GlassVault.submitLiabilityProof(root, totalLiabilities, encryptedDataBlob)`.

### Async Nonce Worker (MATE Special)
To demonstrate the power of the EVVM, we can run multiple instances of this script in parallel, targeting different "shards" of the user base, using Async Nonces to avoid collisions.
