# VPoR Master Implementation Plan

## 1. Executive Summary
VPoR (Verifiable Proof of Reserves) is a permissioned EVVM chain designed to solve the "Black Box" problem of Centralized Exchanges. It uses a "Blockchain-within-a-Blockchain" architecture to provide a transparent, immutable, and privacy-preserving ledger of liabilities, verified against real-world assets via Chainlink.

**Key Objectives:**
1.  **Deploy a Permissioned EVVM Chain (MATE):** A sovereign execution environment for high-throughput liability tracking.
2.  **Integrate Chainlink (CRE/Functions):** To fetch and verify off-chain asset data (BTC, Bank APIs) and bridge it to the VPoR chain.
3.  **Ensure Privacy:** Use ECIES encryption to allow users to verify their specific inclusion in the liability set without exposing their balances publicly.
4.  **Win Hackathon Prizes:** Specifically targeting the **EVVM ($5k)** and **Chainlink ($6k)** tracks at ETHGlobal Buenos Aires.

## 2. Strategic Roadmap (36-Hour Hackathon)

### Phase 1: The Foundation (Hours 0-8)
**Goal:** Establish the core logic and deploy the infrastructure.
-   **Smart Contracts:** Develop `GlassVault.sol` (Core Logic) and `ChainlinkEVVMAdapter.sol`.
-   **Testing:** Verify Merkle Sum Tree logic and Solvency Math using Foundry.
-   **Infrastructure:** Deploy the EVVM Core on Sepolia (Permissioned Mode).

### Phase 2: The Bridge (Hours 8-16)
**Goal:** Connect the Real World to the Virtual Chain.
-   **Chainlink Integration:** Implement the Asset Verification logic.
    -   *Primary Strategy:* Chainlink Functions (Simpler, well-documented).
    -   *Prize Strategy:* Attempt migration to Chainlink Runtime Environment (CRE) if documentation/support permits, to maximize prize eligibility.
-   **Access Control:** Whitelist the Chainlink Adapter as an Executor on the EVVM Chain.

### Phase 3: The Engine (Hours 16-24)
**Goal:** Automate the data flow.
-   **Backend:** Build the "Fisher" (Executor) scripts to:
    1.  Generate dummy user liability data.
    2.  Construct the Merkle Sum Tree.
    3.  Encrypt user proofs (ECIES).
    4.  Submit Liability Proofs to the EVVM Chain.

### Phase 4: The Experience (Hours 24-30)
**Goal:** Make it visible and usable.
-   **Frontend:** Build the "Glass Vault" Dashboard.
    -   Public View: Total Assets vs. Liabilities (Solvency Ratio).
    -   Private View: "Connect Wallet" -> Decrypt Proof -> "Your funds are safe."
-   **Demo Polish:** Create a compelling narrative and video.

### Phase 5: Submission & Pitch (Hours 30-36)
**Goal:** Sell the vision.
-   **Documentation:** Finalize README, Architecture diagrams, and submission text.
-   **Pitch:** Rehearse the "Trust vs. Verification" narrative.

## 3. Critical Success Factors
-   **Async Nonces (EVVM):** Must demonstrate high-throughput updates to justify the custom chain.
-   **Chainlink CRE:** Using CRE is a "nice to have" for the prize, but a working Functions implementation is better than a broken CRE one. We will start with Functions and refactor if time allows.
-   **Privacy:** The "Magic Trick" of decrypting private data on the frontend is the "Wow" moment for the judges.
